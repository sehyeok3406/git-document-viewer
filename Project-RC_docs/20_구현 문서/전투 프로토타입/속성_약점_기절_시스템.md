# 속성 / 약점 / 기절 시스템 명세서

> 문서 상태: 초안  
> 기준 전투 모델: 플레이어 3인 파티 vs 보스 1마리  
> 목적: 속성 적중, 약점 공개, 약점 누적, 보스 기절, 약점 갱신의 판정 규칙을 정의한다

---

# 1. 문서 범위

본 문서는 약점과 기절 판정의 기준 문서다.  
플레이어 행동이 언제 발동되는지는 `플레이어_행동과_스킬.md`에서 정의한다.  
보스가 기절 상태로 전환되는 방식은 `보스_상태_흐름.md`에서 정의한다.  
전투 틱 안에서 판정이 호출되는 순서는 `전투_전체_진행_흐름.md`에서 정의한다.

속성별 피해 배율과 세부 밸런스는 본 문서에서 확정하지 않는다.

---

# 2. 사용 용어

| 용어 | 의미 |
|---|---|
| 속성 | 행동이 가진 판정 태그다. 물리, 화염, 방사능, 중력, 신성, 공허를 우선 사용한다 |
| 약점 슬롯 | 보스가 가진 약점 1칸이다 |
| 미공개 | 플레이어에게 아직 보이지 않는 약점 상태다 |
| 공개 | 속성이 드러나고 누적 공략이 가능한 약점 상태다 |
| 비활성 | 이미 공략이 끝나 다시 사용할 수 없는 약점 상태다 |
| 기절 | 보스 행동이 제한되는 보상 상태다 |

---

# 3. 약점 슬롯 상태

약점 슬롯은 아래 상태를 가진다.

| 상태 | 표시 | 판정 |
|---|---|---|
| hidden | ? | 속성이 맞으면 공개된다 |
| revealed | 속성명과 누적 카운트 | 같은 속성 적중 시 누적된다 |
| disabled | 비활성 | 더 이상 누적되지 않는다 |

각 슬롯은 내부적으로 `element_id`, `required_hits`, `current_hits`, `state`를 가진다.

---

# 4. 속성 판정 조건

약점 판정은 아래 조건을 모두 만족할 때만 실행한다.

- 행동에 `element_id`가 있다
- 행동 데이터의 `weakness_check`가 true다
- 대상이 보스다
- 보스가 전투 불능 상태가 아니다

속성이 없는 행동은 약점 판정을 요청하지 않는다.  
피해량이 0인 속성 행동도 약점 판정을 할지는 [확인 필요]로 둔다.

---

# 5. 약점 공개 흐름

미공개 약점에 맞는 속성 행동이 적중하면 아래 순서로 처리한다.

1. hidden 상태의 약점 슬롯 중 같은 `element_id`를 가진 슬롯을 찾는다
2. 해당 슬롯의 상태를 revealed로 바꾼다
3. `current_hits`를 1 증가시킨다
4. 약점 공개 UI 이벤트를 만든다
5. 약점 누적 UI 이벤트를 만든다
6. 기절 조건을 확인한다

프로토타입 기본값은 `약점 공개 적중도 누적 1회로 계산한다`이다.  
이 방식은 플레이어가 맞는 속성을 찾은 순간의 보상을 더 즉시 느끼게 한다.

---

# 6. 약점 누적 흐름

공개된 약점에 같은 속성 행동이 적중하면 아래 순서로 처리한다.

1. revealed 상태의 약점 슬롯 중 같은 `element_id`를 가진 슬롯을 찾는다
2. `current_hits`를 1 증가시킨다
3. 약점 누적 UI 이벤트를 만든다
4. `current_hits >= required_hits`인지 확인한다
5. 조건을 만족하면 보스 기절 요청을 만든다

같은 속성의 revealed 슬롯이 여러 개라면 가장 먼저 공개된 슬롯에 누적한다.

[확인 필요] 같은 속성 약점 슬롯을 여러 개 허용할지 정해야 한다.  
프로토타입 기본값은 같은 속성 약점 슬롯을 동시에 2개 이상 두지 않는 것이다.

---

# 7. 약점 실패 처리

보스에게 속성 행동이 적중했지만 hidden 또는 revealed 약점과 맞지 않으면 약점 누적은 발생하지 않는다.

실패 정보 표시는 아래 중 하나를 선택해야 한다.

| 옵션 | 처리 |
|---|---|
| 반응 없음 | 별도 메시지를 표시하지 않는다 |
| 약점 아님 표시 | 해당 속성이 약점이 아님을 짧게 표시한다 |
| 기록형 표시 | 시도한 속성을 약점 UI 근처에 실패 기록으로 남긴다 |

프로토타입 기본값은 `약점 아님 표시`를 추천한다.  
플레이어가 속성 탐색 결과를 바로 이해할 수 있기 때문이다.

---

# 8. 기절 발생

약점 슬롯의 `current_hits`가 `required_hits`에 도달하면 보스 기절을 요청한다.

기절 요청이 발생하면 아래 처리를 수행한다.

1. 보스 상태 흐름에 기절 전환을 요청한다
2. 해당 약점 슬롯을 `stun_source`로 기록한다
3. 기절 발생 UI 이벤트를 만든다
4. 전투 로그 이벤트를 만든다

보스가 행동 예고 또는 행동 중이었다면 행동 취소 여부는 `보스_상태_흐름.md`의 규칙을 따른다.

---

# 9. 기절 중 약점 처리

프로토타입 기본값은 `기절 중 약점 누적 정지`이다.  
기절 중 속성 행동은 일반 피해와 기타 효과만 적용하고 약점 공개와 누적은 처리하지 않는다.

이 방식은 약점 공략 구간과 기절 보상 구간을 분리한다.

---

# 10. 기절 종료와 약점 갱신

기절이 종료되면 아래 순서로 약점을 갱신한다.

1. 기절을 발생시킨 약점 슬롯을 disabled로 바꾼다
2. disabled 슬롯의 누적 카운트를 더 이상 표시하지 않는다
3. 남은 후보 속성이 있으면 새 hidden 슬롯을 추가하거나 기존 빈 슬롯에 배치한다
4. 약점 갱신 UI 이벤트를 만든다

보스에게 남은 약점 후보가 없으면 새 약점을 추가하지 않는다.

[확인 필요] 약점 후보가 없을 때 기존 disabled 약점을 다시 활성화할지 정해야 한다.  
프로토타입 기본값은 재활성화하지 않는 것이다.

---

# 11. UI / 로그 이벤트

약점 시스템은 아래 이벤트를 화면과 로그에 전달한다.

| 이벤트 | 발생 조건 |
|---|---|
| weakness_revealed | hidden 약점이 공개됨 |
| weakness_hit | revealed 약점 누적 증가 |
| weakness_miss | 약점이 아닌 속성 적중 |
| boss_stunned | 기절 조건 달성 |
| weakness_disabled | 기절 종료 후 약점 비활성화 |
| weakness_refreshed | 새 hidden 약점 추가 |

이벤트의 표시 위치와 우선순위는 `화면_구성_명세.md`를 따른다.
